git 的.ignore文件的使用

先修改Project的.ignore文件

  1. 先将项目所有的代码都从git里清空掉 (对应到编辑器/IDE就是: 已经修改过的代码行前面绿色标记消失了)

  2. 提交

git merge master:

在master分支上, 把Alice切出分支的那个祖先基点以后的提交合并成一次提交(dev1), 提交至dev分支上.
这样做的副作用就是: 在Alice的dev分支上多出了一次提交记录(dev1)

这次的提交记录并不是Alice完成的, 是git自动提交的,

git rebase master :

(它的原理正如它的 字面意思,别想太多, 就是更新基点)

首先, git会把dev分支的所有commit都取消掉;
其次，把上面的操作临时保存成 patch 文件，存在 .git/rebase 目录下;
然后, 更新dev分支的基点为master分支最新;
最后, 把上面保存的 patch 文件应用到 feature1 分支上;

Warnning:
在 rebase 的过程中, 也许会出现冲突 conflict.
在这种情况，git 会停止 rebase 并会让Alice去解决冲突. 在解决完冲突后,用 git add 命令去更新这些内容, 这时Alice不必使用git commit 来提交, 只需要使用git rebase –continue, 这样git会继续使用余下没用使用的patch补丁文件.

git-rebase 存在的价值是：对一个分支做「变基」操作。

  1. 当Alice在一个过时的分支上面开发的时候，执行 rebase 以此同步 master 分支最新变动；
  2. 假如Alice要启动一个放置了很久的并行工作，现在有时间来继续这件事情，很显然这个分支已经落后了。这时候需要在最新的基准上面开始工作，所以 rebase 是最合适的选择。

副作用: 使用git rebase只会改变Alice的基点, 其他人在dev的基点仍然是旧的, 所以这是一个危险的操作, 会让其他人丢失记录;

比较:

merge是在所有的提交之后更新, 把别人的提交拿过来应用到自己的分支上
rebase 是直接更新所有提交的基点.
