session,cookie,token的理解

Posted on 2019-11-13

session,cookie,token其实都是session(会话)的各种实现(并不止于三种)
session和cookie都是session, cookie是session的一种实现.就像interface一样.
个人感觉Token也是session的一种实现.

首先, 来看看解决的问题是什么:

http的这一次的请求和上一次的请求是没有任何关系的,互不认识的,没有关联的.这就是传说中的http是无状态的
好处是处理起来快速, 坏处就是想要把这次请求和上次请求关联起来就要采取手段, 手段就是:session,cookie和token.

session里面 建议且主要 是存储用户相关的信息, 比如用户ID,上次登录时间,过期时间,登录次数 等.

那么, 这三个有啥区别呢?

简单来说就是:(主要)
session 是把状态存储在服务器中, 存储在数据库或者文件中(默认存在文件中,也可以在数据库/内存中)
cookie 是把状态存储在客户端, 存储在用户硬盘的一个文件夹
session可以存储复杂对象, cookie只能存储String,且是ASCII字符.

Token又是解决了什么痛点呢?

  • 保存session在服务器, 如果A机器挂了, 那么一批用户的登录状态都挂了. 如果A机器很忙,B机器来处理,B机器没有这个用户的登录状态,这样也不行.
    如果把session存在一个共同的机器/集群上,那么存储session的机器或者集群挂了呢?
  • 保存cookie在客户端, 如果被别人复制了, 那就不好了.

所以说, 要想一个办法, 既不保存在服务器端,又不保存在用户硬盘. Token就是这样一种方法.

既然解决的问题是 服务器要把这次的请求和上次的请求关联起来
那么, 在客户登录的时候生成一个登录信息表,保存在服务器上.也就是session
就给客户端一个ID, 让客户端每次请求的时候拿着这个id来请求服务器,这个id就是sessionid
基本思路就是这样的.

因为是 服务器要把这次的请求和上次的请求关联起来, 那么只用cookie是行不通的,因为cookie是存在客户端,且只存ASCII字符串.只用Session倒是可以.

第一种: 只用session: (也就是在浏览器禁用cookie的情况下)

  • URL重写: 客户端请求的时候在URL中加上这个sessionid
  • 隐藏表单字段: 在请求的from表单中自动添加一项参数,一起发送给服务器

第二种: 其实呢,session基本不会单独使用, 一般都是session + cookie.

大致过程是这样的:

Step1:客户端把用户ID和密码等登录信息放入报文的实体部分,通常是以POST方法把请求发送给服务器

Step2:服务器会发放用以识别用户的SessionID.通过验证从客户端发送过来的登录信息进行身份认证,然后把用户的认证状态与SessionID 绑定后记录在服务器端.
向客户端返回响应时,会在头部字段Set-Cookie内写入SessionID (Set-Cookie: PHPSESSID=028a8c…)
你可以把SessionID 想象成一种用以区分不同用户的等位号.然而,如果SessionID 被第三方盗走,对方就可以伪装成你的身份进行恶意操作了.
因此必须防止SessionID 被盗,或被猜出.为了做到这点,SessionID 应使用难以推测的字符串,且服务器端也需要进行有效期的管理,保证其安全性.
另外,为减轻跨站脚本攻击(XSS)造成的损失,建议事先在Cookie 内加上httponly 属性.
Cookie 的HttpOnly 属性是Cookie 的扩展功能,它使JavaScript 脚本无法获得Cookie. cookie也是有标准格式的.但可以不一定按照标准来.

响应头里面 Set-Cookie: name=value; HttpOnly

Step3:客户端接收到从服务器端发来的SessionID 后,会将其作为Cookie 保存在本地.下次向服务器发送请求时,浏览器会自动发送Cookie,
所以SessionID 也随之发送到服务器.服务器端可通过验证接收到的SessionID 识别用户和其认证状态.
请求头里面 cookie: sid = SessionID

第三种: Token是怎么搞的呢?

( 前面说到了token解决的痛点,token是不储存在客户端或者服务器的.那么就是经典操作:时间换空间 )

可是如果服务器不保存这些SessionID, 服务器没法验证客户端发送的SessionID是不是有效的合法的.
如果服务器不去验证,服务器就不知道他们是不是合法登录的用户, 这样也是不行的.
所以说, 关键点就是验证

比如说:
Alice已经登录了系统,服务器给他发一个token,里边的payload(负载)包含了Alice的userId,
下一次Alice再次通过Http请求访问服务器的时候,把 token 放在请求的 header 里面的cookie(JSON Web Token).
不过这和SessionID没有本质区别,任何人都可以可以伪造.

那就对数据做一个签名: 把这个签名和数据一起作为token,由于密钥别人不知道,就无法伪造token了.
服务器不保存这个token,当请求来的时候,服务器再对用户的userID数据做一次计算,得到token2,
如果token2与传过来的token相同,就表明Alice已经登录过了.如果不相同就说明数据被篡改过.

Token中的数据是明文保存的（虽然我会用Base64做下编码,但那不是加密, 还是可以被别人看到的, 所以我不能在其中保存像密码这样的敏感信息.
如果token被偷走了那没办法.就像session被偷走了一样.
(为什么密钥别人没法通过暴力算法算出来,搜索关键字: 活板门函数 单向函数 单向陷门函数)

  • 如果cookie设置为 HttpOnly, JS是无法拿到的,会跳过这个属性.所以说token一般会放在httpOnly里面
  • token不仅可以存在cookie中,也可以存在localStorage中.
  • token过期设计, 可以见我的JWT文章

警告: 以下内容非常重要
经常有些项目/系统/project(管你叫啥),生成了一个字符串放在了redis,自称是token,但是每次请求还是要去redis取userid.
这不是token, 这是session.
前面已经说了 token解决的痛点:是不储存在服务器的. 这是SessionId.
