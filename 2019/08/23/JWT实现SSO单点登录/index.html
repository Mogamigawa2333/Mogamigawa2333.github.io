JWT 实现 SSO 单点登录

Posted on 2019-08-23

概念解释:

  • JWT: JSON Web Token, 是一个标准/规范. 在每次请求时, 把 token 放在请求的 header 里面.
  • SSO: Single Sign On, 在多个应用的系统中, 用户只登录一次,就可以访问所有相互信任的系统.

JWT 的工作原理:

JWT 的组成:

payLoad 的其他保留关键字:

  • iss: The issuer of the token, token 是给谁的
  • sub: The subject of the token, token 主题
  • exp: Expiration Time. token 过期时间,Unix 时间戳格式
  • iat: Issued At. token 创建时间, Unix 时间戳格式
  • jti: JWT ID. 针对当前 token 的唯一标识

JWT 是如何验真的?

Server 在接收到请求后, 把 http Header 里面 token 取出
然后用 密钥 把 token 的 头部.载荷 进行签名. 得到新的签名.
最后将 http 里面的签名和新的签名进行比较.

如果是一致的 –> token 没有被篡改,
相反不是一致 –> token 已经被篡改.

业务逻辑设计:

登录有两种:

  1. 既然 token 的载荷里面有 过期时间 exp 这个字段, 那么只要后台验真即可, 只要不到有效期就都可以使用.
    优点是设计简单且后台不用存储,缺点是如果在有效期内有人恶意刷接口,后台将无能为力.

  2. 使用 token + redis 的方式: 访问时判断 redis 里是否有该 token，如果有，则加载该用户信息完成登录。服务需要存储下发的每个 token 及对应的 value，维持其过期时间.
    优点是随时可以删除某个 token，阻断该 token 继续使用. 而且 token 可以使用随机数生成不一定使用 JWT.

退出:

有以下几个方法可以做到失效 JWT token：

  • 将 token 存入 DB（如 Redis）中，失效则删除；但增加了一个每次校验时候都要先从 DB 中查询 token 是否存在的步骤，而且违背了 JWT 的无状态原则（这不就和 session 一样了么？）。
  • 维护一个 token 黑名单，失效则加入黑名单中。
  • 在 JWT 中增加一个版本号字段，失效则改变该版本号。
  • 在服务端设置加密的 key 时，为每个用户生成唯一的 key，失效则改变该 key

SSO 设计:

目的:

就像 SpringCloud 那种的分布式服务, 拥有很多的微服务, 让用户在一次登录以后就可以访问同一系统下的所有的服务.
如: xxx.taobao.com
ooo.taobao.com
aaa.taobao.com
login.taobao.com
用户只要访问 login.taobao.com 登录, 就可以访问其他服务.

解决方案:

在放 token 的 cookie 中, 添加一个 domian 属性,注意 domain 必须设置为一个点加顶级域名，即.taobao.com 这样的，taobao.com和\*.taobao.com 就都可以接受到这个 Cookie，并获取 JWT 了[其实有一个 CAS(集中式授权认证服务),CAS 知道 JWT 的密钥,这样其他服务就不用都需要知道 JWT 密钥了]

既然 token 有有效期,而且前端是无状态的, 那么前端是怎样存储 token 的?

安全性建议:

  1. 使用 https

  2. 使用 http-only 的 cookie，针对 sid 和 jwt

  3. 管理好密钥
