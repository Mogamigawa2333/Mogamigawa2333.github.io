接口签名算法

Posted on 2020-02-25

设计接口记住要有验证处理,要么Token要么Sign

接口添加签名,我有两种方式:

传统方式:

传统方式是使用表单提交的方式来请求, 使用json的方式来响应;
请求的header的Content-type键的值为application/x-www-form-urlcode.
从content-type的值可以看出参数是使用form方式提交,里面的参数使用的urlcode的方式进行编码.

请求过程是这样的:

  1. 排除sign参数
  2. 设所有发送或者接收到的数据为集合M，将集合M内非空参数值的参数按照参数名ASCII码从小到大排序（字典序），使用URL键值对的格式（即key1=value1&key2=value2…）拼接成字符串stringA
  3. 在stringA最后拼接上KEY得到stringSignTemp字符串（即stringSignTemp = stringA + &key=KEY），并对stringSignTemp进行MD5运算，再将得到的字符串所有字符转换为大写，得到sign值signValue
  4. 形成新的接口: 以下是示例
    https://api.test.com/dosomething?sign=C0F1EB5A06DF4A46207645606B6156C8&timestamp=1582275984&param1=value1&param2=value2
  5. 为了防止接口被多次调用, 可以添加timestamp参数

一种新的签名方式:

请求和响应都是JSON方式:

  1. 把所有参数组装为JSON对象或者JsonArray对象 jsonA

    例如: {"key1":"value1","key2":"value2"}

  2. 将jsonA转为Base64URL编码的字符串stringA

    例如:YHsia2V5MSI6InZhbHVlMSIsImtleTIiOiJ2YWx1ZTIifQ

  3. 对stringA和密钥使用HmacSHA256做摘要, 得到签名字符串signString,

    例如: 密钥为qwerty, 生成的签名为cee282508e9131665a0987eb5c49855b6bac2e56beb0e651e73cec8158b1e891

  4. 构造一个新的JSON对象,里面有两个键, 一个为data, 另一个为sign.

    easyjoy的值为jsonA对象, sign的值为signString.

    例如最后构造出来的json为: {"data":{"key1":"value1","key2":"value2"},"sign":"cee282508e9131665a0987eb5c49855b6bac2e56beb0e651e73cec8158b1e891"}

  5. 验签的时候,只对data的值进行验证.

  6. 由于是单个系统间的交互, 如果验签通过则证明密钥正确, 密钥正确则可证明身份, 无需token

关于签名算法的说明:

MD2
MD5
SHA-1
SHA-256
SHA-384
SHA-512
HmacMD5
HmacSHA1
HmacSHA256
HmacSHA384
HmacSHA512

以上这些都是摘要算法, 其中HmacSHA常见的是其缩写形式HS.
MD5之类的都是直接摘要, 所以对于MD5来说,密钥要拼在要加密的字符串的后面.
其实对于MD5来说根本没有密钥这一说法.

HMAC,全称为:Hash Message Authentication Code, 中文名”散列消息鉴别码”.
主要是利用哈希算法, 以一个密钥和一个消息为输入,生成一个消息摘要为输出.
一般的,消息鉴别码用于验证传输于连个共同享有一个密钥的单位之间的消息,HMAC可以与任何迭代的散列函数捆绑使用,MD5和SHA-1就是这种散列函数,
HMAC还可以使用一个用于计算和用确认消息鉴别值的密钥.
